# PRD-001: Order Lifecycle And Operations

## Problem

`telegram_shop_bot` уже закрывает базовую shopping-задачу: пользователь может войти в бота, пройти по каталогу, открыть карточку товара и собрать корзину. Однако продукт пока не закрывает следующий обязательный шаг покупательского пути: из корзины нельзя оформить заказ, оплатить его, отслеживать исполнение и получить поддержку после оформления.

В результате текущий baseline создает интерес к покупке, но не доводит пользователя до завершенной транзакции. Для бизнеса это означает ручную дообработку заказов вне бота, потерю конверсии между выбором товара и оплатой, а также отсутствие единого операционного контура для работы с заказами, статусами и коммуникацией с покупателем.

Эта инициатива берет в работу delta относительно [`../domain/problem.md`](../domain/problem.md): превращение каталога и корзины в end-to-end commerce flow внутри Telegram с минимально необходимыми backoffice-возможностями.

## Users And Jobs

| User / Segment | Job To Be Done | Current Pain |
| --- | --- | --- |
| `buyer` | Выбрать товары, оформить заказ и понять, что происходит дальше | После сборки корзины нет checkout flow, оплаты и статусов заказа |
| `store-operator` | Получать новые заказы, менять их статус и коммуницировать с покупателем | Нет встроенного рабочего контура для обработки заказов в Telegram |
| `store-admin` | Управлять каталогом и операционными настройками без отдельной админки | Нет прав, инструментов управления товарами и массовых коммуникаций |

## Goals

- `G-01` Покупатель может завершить путь `каталог -> корзина -> оформление заказа -> подтверждение` полностью внутри Telegram.
- `G-02` Система хранит заказ как отдельную сущность с номером, контактными данными, адресом, составом и текущим статусом.
- `G-03` Бот поддерживает как минимум два способа расчета: онлайн-оплата и оплата при получении.
- `G-04` Покупатель может в любой момент запросить статус заказа и получить актуальное состояние без участия менеджера.
- `G-05` Оператор или администратор может видеть новые заказы и обновлять их статус через Telegram-интерфейс с учетом ролей.

## Non-Goals

- `NG-01` Инициатива не описывает детальную техническую реализацию платежного провайдера, CRM-коннектора или админского интерфейса на уровне architecture/design.
- `NG-02` Инициатива не требует в первой версии полноформатного e-commerce backoffice уровня ERP/OMS.
- `NG-03` Инициатива не гарантирует запуск всех wishlist-возможностей из `PROJECT.md`; отзывы, операторский чат и глубокая CRM-интеграция могут остаться последующими фазами.
- `NG-04` Инициатива не должна ломать существующий baseline каталога, карточки товара и корзины.

## Product Scope

Инициатива расширяет продукт от pre-purchase browsing к полному циклу заказа и его сопровождения в Telegram.

### In Scope

- Оформление заказа из корзины со сбором обязательных контактных данных.
- Подтверждение заказа пользователем перед финальной фиксацией.
- Генерация уникального номера заказа и сохранение заказа в PostgreSQL.
- Отражение способа оплаты и статуса оплаты в заказе.
- Отдельный пользовательский сценарий проверки статуса заказа.
- Оповещения покупателю при изменении статуса заказа.
- Базовый операторский контур для просмотра новых заказов и смены статуса.
- Базовое разграничение ролей для пользовательских и административных действий.
- Управление товарами и категориями через Telegram как операционная capability.
- Массовые рассылки и базовая статистика как административные возможности.

### Out Of Scope

- Web checkout, web admin panel или любая обязательная вне-Telegram UI-поверхность.
- Сложная логистика, расчет доставки по внешним API и multi-warehouse сценарии.
- Программа лояльности, промокоды, персональные рекомендации.
- Полноценный helpdesk/workflow engine для операторов.
- Обязательная синхронизация с конкретной CRM до выбора интеграционного контракта.

## UX / Business Rules

- `BR-01` Telegram остается единственной обязательной пользовательской поверхностью; все основные сценарии должны укладываться в сообщения и inline/reply keyboard.
- `BR-02` Checkout не должен очищать и терять корзину до явного подтверждения заказа пользователем.
- `BR-03` Заказ должен иметь устойчивый уникальный номер, пригодный для коммуникации с покупателем и оператором.
- `BR-04` Статусы заказа и оплаты должны храниться в PostgreSQL и быть доступны для повторного чтения между сессиями.
- `BR-05` Пользовательские уведомления о смене статуса должны отправляться автоматически и ссылаться на конкретный заказ.
- `BR-06` Роли должны ограничивать административные операции; обычный покупатель не должен получать доступ к управлению заказами, товарами и рассылками.
- `BR-07` Оплата при получении должна поддерживаться даже если онлайн-оплата временно недоступна.
- `BR-08` Если интеграция с внешней системой отсутствует или деградирует, базовый процесс оформления и обработки заказа внутри бота должен оставаться работоспособным.

## Success Metrics

| Metric ID | Metric | Baseline | Target | Measurement method |
| --- | --- | --- | --- | --- |
| `MET-01` | Пользователь завершает оформление заказа внутри Telegram | Сейчас путь обрывается на корзине | Пользователь проходит путь `корзина -> ввод контактов -> подтверждение -> созданный заказ` без ручной дообработки | Acceptance flow и integration tests по checkout/order services |
| `MET-02` | Заказ имеет отслеживаемый lifecycle | Отдельная сущность заказа и статусный поток отсутствуют | Каждый оформленный заказ получает номер, статус заказа и статус оплаты | Проверка БД, acceptance-сценарии и ручная валидация статусов |
| `MET-03` | Покупатель получает прозрачность по исполнению заказа | Статус заказа нельзя запросить в боте | Пользователь может открыть статус заказа и получить актуальное состояние по своему заказу | End-to-end сценарий user flow |
| `MET-04` | Оператор может обработать заказ без внешнего интерфейса | Работа с заказом вне бота не стандартизирована | Новый заказ виден оператору, а смена статуса приводит к обновлению в системе и уведомлению покупателя | Ручной сценарий admin/operator flow |
| `MET-05` | Каталог и корзина не деградируют при расширении продукта | Существующий baseline уже реализован и ценен | Базовые сценарии каталога, карточки и корзины продолжают проходить без регрессий | Regression tests по текущим workflows |

## Risks And Open Questions

- `RISK-01` Scope из `PROJECT.md` слишком широк; без фазирования инициатива может распасться на слабо связанные задачи и затянуть delivery.
- `RISK-02` Выбор платежной интеграции и модель обработки ошибок оплаты пока не определены.
- `RISK-03` Административные возможности через Telegram могут быстро упереться в UX-ограничения chat-интерфейса.
- `RISK-04` Массовые рассылки и статусные уведомления требуют аккуратной политики, чтобы не превратить бота в канал спама.
- `OQ-01` Какие статусы заказа считаются каноническими и нужен ли отдельный статусный граф для возвратов/отмен?
- `OQ-02` Должен ли пользователь иметь доступ к истории всех своих заказов или только к последнему/конкретному заказу?
- `OQ-03` Какие административные возможности обязательны для первой фазы: только обработка заказов или также управление каталогом и рассылки?
- `OQ-04` Является ли CRM-интеграция частью первой коммерчески ценной версии или ее нужно вынести в отдельный PRD/phase?

# UC-006: Manage Catalog

## Goal

Администратор управляет структурой каталога и карточками товаров внутри Telegram-бота без прямого доступа к БД или внешнему backoffice.

## Primary Actor

Администратор магазина.

## Trigger

Пользователь с ролью `admin` нажимает entrypoint управления каталогом в главном меню Telegram-бота.

## Preconditions

- Пользователь зарегистрирован в таблице `users`.
- `users.role` пользователя равен `admin`.
- PostgreSQL содержит или готов принять сущности `categories` и `products`.
- Telegram UI остается единственной обязательной поверхностью операционного сценария.

## Main Flow

1. Администратор открывает сценарий управления каталогом из главного меню.
2. Бот показывает список категорий текущего уровня и доступные действия управления.
3. Администратор открывает нужную категорию или создаёт новую.
4. Бот принимает пошаговый ввод и сохраняет create/update действие в PostgreSQL.
5. Администратор при необходимости открывает листовую категорию, создаёт или редактирует товар.
6. Система сохраняет изменения и возвращает обновлённый экран admin workflow.

## Alternate Flows / Exceptions

- `ALT-01` Если на выбранном уровне нет категорий или товаров, бот показывает explicit empty-state и доступные create actions.
- `ALT-02` Если категория пустая, администратор может удалить её и бот возвращает обновлённый список.
- `EX-01` Если пользователь без роли `admin` пытается открыть сценарий, бот показывает безопасный отказ в доступе.
- `EX-02` Если категория непустая, её удаление блокируется безопасным сообщением без изменения данных.
- `EX-03` Если create/update/delete завершается инфраструктурной ошибкой, бот показывает безопасное сообщение без технических деталей.

## Postconditions

- После успешного сценария изменения категорий и товаров сохранены в PostgreSQL.
- Buyer catalog читает актуальные данные без отдельной синхронизации.
- Неавторизованный доступ не раскрывает admin workflow и не меняет каталог.

## Business Rules

- `BR-01` Только роль `admin` имеет доступ к сценарию управления каталогом.
- `BR-02` Любая реализация сценария должна оставаться в Telegram chat UI и не требовать web admin panel.
- `BR-03` Категория удаляется только если у неё нет дочерних категорий и товаров.
- `BR-04` Товар создаётся только в листовой категории.
- `BR-05` Один admin action приводит к одному итоговому ответу: обновлённому экрану данных, prompt следующего шага или безопасному сообщению об ошибке.

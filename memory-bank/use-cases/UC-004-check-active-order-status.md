# UC-004: Check Active Order Status

## Goal

Покупатель может в любой момент открыть список своих активных заказов и увидеть их актуальные статусы без обращения к оператору.

## Primary Actor

Покупатель интернет-магазина.

## Trigger

Пользователь нажимает кнопку `Статус заказа` в главном меню Telegram-бота.

## Preconditions

- Пользователь зарегистрирован в таблице `users`.
- Заказы пользователя сохранены в PostgreSQL и содержат актуальный `status`.
- Telegram UI остается единственной пользовательской поверхностью сценария.

## Main Flow

1. Пользователь нажимает `Статус заказа`.
2. Бот читает активные заказы пользователя из PostgreSQL.
3. Бот показывает список активных заказов с номером заказа и человекочитаемым статусом.
4. Пользователь понимает, какие его заказы еще находятся в исполнении.

## Alternate Flows / Exceptions

- `ALT-01` Если у пользователя нет активных заказов, бот показывает понятное сообщение об их отсутствии.
- `EX-01` Если чтение заказов завершается инфраструктурной ошибкой, бот показывает безопасное сообщение об ошибке без технических деталей.

## Postconditions

- После успешного сценария пользователь видит актуальный статус каждого активного заказа.
- Сценарий не изменяет заказы, корзину и профиль пользователя.

## Business Rules

- `BR-01` Сценарий показывает только заказы текущего пользователя.
- `BR-02` В список активных заказов не входят terminal-state заказы со статусами завершения или отмены.
- `BR-03` Пользовательский текст должен использовать понятные русскоязычные названия статусов, даже если в БД статус хранится как техническое значение; для текущего lifecycle это как минимум `Создан`, `Оплачен`, `Собран`, `Передан в доставку`, `Получен`, `Отменен`.
- `BR-04` Один пользовательский action приводит к одному итоговому ответу: списку активных заказов, сообщению об отсутствии заказов или безопасной ошибке.

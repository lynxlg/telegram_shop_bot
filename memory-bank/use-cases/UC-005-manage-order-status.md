# UC-005: Manage Order Status

## Goal

Оператор или администратор открывает активные заказы внутри Telegram-бота и переводит конкретный заказ в актуальный статус исполнения без внешнего интерфейса.

## Primary Actor

Оператор магазина или администратор магазина.

## Trigger

Пользователь с ролью `operator` или `admin` нажимает entrypoint управления заказами в главном меню Telegram-бота.

## Preconditions

- Пользователь зарегистрирован в таблице `users`.
- `users.role` пользователя равен `operator` или `admin`.
- В PostgreSQL сохранены заказы, содержащие `order_number` и `status`.
- Telegram UI остается единственной обязательной поверхностью операционного сценария.

## Main Flow

1. Оператор открывает сценарий управления заказами из главного меню.
2. Бот читает активные заказы из PostgreSQL и показывает список доступных заказов.
3. Оператор выбирает конкретный заказ.
4. Бот показывает карточку заказа с текущим статусом, контактами и действиями смены статуса.
5. Оператор выбирает новый статус заказа.
6. Система сохраняет новый статус заказа в PostgreSQL.
7. После успешного обновления бот отправляет покупателю уведомление с номером заказа и новым статусом.
8. Бот показывает обновленную карточку заказа и позволяет вернуться к списку активных заказов.

## Alternate Flows / Exceptions

- `ALT-01` Если активных заказов нет, бот показывает понятное сообщение об их отсутствии.
- `ALT-02` Если после обновления заказ переходит в terminal-state, он больше не попадает в список активных заказов.
- `ALT-03` Если оператор выбирает уже сохраненный статус, заказ остается без изменений и buyer notification не отправляется.
- `EX-01` Если пользователь без роли `operator`/`admin` пытается открыть сценарий, бот показывает безопасный отказ в доступе.
- `EX-02` Если чтение или обновление заказа завершается инфраструктурной ошибкой, бот показывает безопасное сообщение без технических деталей.
- `EX-03` Если отправка buyer notification завершается ошибкой после commit, статус заказа остается обновленным, а ошибка только логируется.

## Postconditions

- После успешного сценария статус выбранного заказа обновлен в PostgreSQL.
- После фактической смены статуса покупатель получает Telegram-уведомление с номером заказа и новым статусом.
- После перевода заказа в `Получен` или `Отменен` он исчезает из списка активных заказов.
- Неавторизованный доступ не меняет заказ и не раскрывает operator workflow.

## Business Rules

- `BR-01` Только роли `operator` и `admin` имеют доступ к сценарию управления заказами.
- `BR-02` Канонический пользовательский набор статусов для этого сценария: `Создан`, `Оплачен`, `Собран`, `Передан в доставку`, `Получен`, `Отменен`.
- `BR-03` Любая реализация сценария должна использовать один и тот же status mapping для operator UI и buyer-facing просмотра статуса заказа.
- `BR-04` Terminal-state заказы не считаются активными и не должны возвращаться в список активных заказов.
- `BR-05` Buyer notification отправляется только после фактической смены статуса заказа.
- `BR-06` Ошибка доставки buyer notification не отменяет уже сохраненный status update.
- `BR-07` Один operator action приводит к одному итоговому ответу: списку заказов, карточке заказа, обновленной карточке или безопасному сообщению об ошибке/отказе.

# Frontend

В проекте нет отдельного web frontend или mobile app. Пользовательский интерфейс целиком реализован как Telegram chat UI через `aiogram`, поэтому документ фиксирует только правила для сообщений, reply keyboard, inline keyboard и callback-навигации.

## UI Surfaces

- `customer chat UI`
  Код: `app/handlers/*`, `app/keyboards/*`, `app/callbacks/*`, `app/services/*_text.py`
  Стек: `Aiogram 3`, Telegram Bot API, reply keyboard и inline keyboard
  Boundary: handler получает update, вызывает сервисы, затем отрисовывает сообщение и клавиатуру
  Owner for design decisions: application layer этого репозитория; внешнего design system или frontend-пакета нет

- `admin chat UI`
  Код: `app/handlers/admin_catalog.py`, `app/handlers/operator_orders.py`, `app/keyboards/admin_catalog.py`, `app/keyboards/operator_orders.py`, `app/callbacks/admin_catalog.py`, `app/callbacks/operator_orders.py`
  Стек: `Aiogram 3`, Telegram Bot API, role-gated reply keyboard и inline keyboard
  Boundary: административные сценарии остаются внутри Telegram; отдельный web backoffice по-прежнему отсутствует
  Owner for design decisions: application layer этого репозитория; любые дальнейшие admin flows должны оставаться совместимыми с ограничениями chat UI

## Component And Styling Rules

- Единой design system нет; переиспользуемыми UI-компонентами считаются keyboard builders и callback schemas.
- Главное меню на постоянных действиях оформляется через `ReplyKeyboardMarkup`; навигация внутри сценария и действия над сущностями оформляются через `InlineKeyboardMarkup`.
- Текст экрана, если он собирается из нескольких полей доменной сущности, должен выноситься в `app/services/catalog_text.py`, `app/services/cart_text.py` или аналогичный text-builder слой, а не собираться хаотично по handlers.
- Новые callback payloads должны оформляться отдельным типом в `app/callbacks/*`, чтобы парсинг и генерация кнопок использовали один и тот же контракт.
- Telegram UI не поддерживает проектные CSS/spacing/token rules; вместо этого важны краткость текста, предсказуемый порядок кнопок и стабильные названия действий.

## Interaction Patterns

- Основной паттерн — event-driven chat flow: пользователь отправляет команду или нажимает кнопку, бот отвечает новым сообщением или редактированием текущего сообщения.
- Для каталога и корзины canonical поведением является обновление текущего inline-экрана через `edit_text`/`edit_media`, а при невозможности такого обновления — fallback на отправку нового сообщения с удалением старого, если это безопасно.
- Для глобального входа в сценарий используются текстовые кнопки главного меню (`Каталог`, `Корзина`, role-gated operational/admin entrypoints), а не свободный пользовательский текст.
- Один пользовательский action должен приводить к одному итоговому ответу: экрану данных или сообщению об ошибке. Промежуточные частично заполненные ответы не считаются допустимым UI.
- Если новая feature требует многошаговый ввод данных, её нужно проектировать под ограничения Telegram-сессии: короткие шаги, явные кнопки возврата и отсутствие сложных форм, завязанных на произвольную верстку.

## Canonical Screen Contracts

- Каталог начинается из reply-кнопки `Каталог` и ведет по дереву категорий до списка активных товаров листовой категории.
- Карточка товара обязана показывать `name` и `price`; при отсутствии `description` и `attributes` используются тексты `Описание отсутствует.` и `Характеристики не указаны.`
- Если у товара есть `image_url`, карточка товара показывается как media-screen с тем же текстовым контрактом; отсутствие `image_url` не должно блокировать открытие карточки.
- Кнопка `Назад` в карточке товара должна возвращать пользователя на предыдущий экран каталога, а не в произвольную точку навигации.
- Добавление товара в корзину инициируется из карточки товара кнопкой `В корзину`.
- Экран корзины открывается из reply-кнопки `Корзина` и показывает позиции, их количество, стоимость позиции и итоговую сумму.
- Для действий над позицией корзины используются inline-кнопки `+`, `-`, `Удалить`; после действия пользователь видит обновленный экран корзины или `Корзина пуста.`
- Экран admin catalog открывается только для роли `admin` из отдельной reply-кнопки и показывает иерархию категорий, список товаров текущего раздела и inline-действия CRUD без выхода в web UI.
- Admin create/edit flow допускает пошаговый текстовый ввод через FSM, но итоговым пользовательским ответом после каждого шага подтверждения остаётся один prompt или один обновлённый admin screen.

## Localization

- Отдельного i18n-слоя сейчас нет, но пользовательские строки уже централизованы в [app/ui_texts.json](../../app/ui_texts.json) и читаются через [app/ui_text.py](../../app/ui_text.py).
- Это означает, что изменение copy остаётся изменением кода репозитория, но основной owner текстов — JSON-словарь UI, а не хаотичные inline-строки по handlers.
- Новые тексты и кнопки нужно добавлять в `app/ui_texts.json`, а затем читать через `get_ui_text()` / `format_ui_text()`, сохраняя текущий русскоязычный стиль.
- Если проекту потребуется мультиязычность, нужно сначала определить canonical owner каталогов переводов и migration rule, а не смешивать `ui_texts.json` с произвольными inline-строками.
